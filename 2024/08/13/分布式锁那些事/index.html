分布式锁那些事

之前皓哥在公司每周的技术分享会上分享了分布式锁相关的内容，项目里也用到了分布式锁，基于分享的内容和实践对分布式锁做了一个系统性的学习，这里做一个总结。

为什么需要分布式锁

如果我们的程序需要对某些共享资源 进行并发访问，在单机环境下，我们可以使用 synchronized、ReentrantLock 等锁来保证线程安全，但是在分布式微服务环境下，多个服务实例之间无法共享锁，所以需要引入分布式锁来保证分布式环境下的线程安全。

案例一：订单状态的修改

在商城系统中，用户下单后，订单状态会从未支付变为已支付，这个过程中可能会有多个服务实例同时处理订单状态的修改，如果不加锁，可能会导致订单状态异常，比如用户支付成功后，订单状态没有及时修改，导致用户再次支付，这时候就需要使用分布式锁来保证订单状态的修改是线程安全的。

案例二：限制自由访问

在 MegaEase Cloud 云平台中，需要同时访问一个或多个主机，执行安装监控、部署或者启停中间件等操作。为了保证操作安全性，我们需要保证某个主机在同一时刻只能执行一个任务，这时候就需要使用分布式锁来保证对主机的互斥访问。

分布式锁的要点

互斥性：同一时刻只能有一个客户端持有锁。
活性：无死锁，即使持有锁的客户端崩溃或者网络异常，锁也能被释放。
容错性：只要大部分节点正常运行，客户端就可以获取和释放锁。
解铃还需系铃人：只有加锁的客户端才能释放锁。

1. 互斥性

分布式锁的第一个要点是互斥性，即同一时刻只能有一个客户端持有锁。

2. 无死锁

分布式锁的第二个要点是无死锁，即即使持有锁的客户端崩溃或者网络异常，锁也能被释放。

3. 容错性

分布式锁的第三个要点是容错性，即只要大部分节点正常运行，客户端就可以获取和释放锁。

4. 解铃还需系铃人

分布式锁的第四个要点是解铃还需系铃人，即只有加锁的客户端才能释放锁。

5. 性能

分布式锁的第五个要点是性能，即加锁和释放锁的性能要好。

6. 重入性

分布式锁的第六个要点是重入性，即同一个客户端可以多次加锁，但是要对应释放锁。

7. 时效性

分布式锁的第七个要点是时效性，即加锁和释放锁的操作要有时效性，避免死锁。

8. 高可用

分布式锁的第八个要点是高可用，即分布式锁要有高可用性，避免单点故障。

9. 高性能

分布式锁的第九个要点是高性能，即分布式锁的性能要好，避免成为系统的性能瓶颈。

10. 易用性

分布式锁的第十个要点是易用性，即分布式锁的使用要简单，避免出现使用错误。

分布式锁的实现方式

2. 基于单机 Redis 的实现

分布式锁最常见的实现应该就是基于单节点的 Redis 实现了，Redis 本身提供了 setnx 和 expire 两个命令，可以帮助我们实现分布式锁。

  • resource_name：锁的名称，可以是业务名称，也可以是 key。
  • my_random_value：锁的值，可以是随机值，也可以是客户端 ID。
  • NX：只有当 key 不存在时才设置 key 的值。
  • PX 30000：设置 key 的过期时间为 30000 毫秒。

单机实现的问题

因为只有一个 Redis 节点，所以 Redis 单机实现的分布式锁有以下几个问题：

  • 单点故障：如果 Redis 节点宕机，那么所有的锁都会失效。
  • 锁不可重入：如果一个客户端加锁后，又再次加锁，那么第二次加锁会失败。
  • 锁不可续约：如果一个客户端加锁后，没有在过期时间内释放锁，那么锁会失效。
