调整发送窗口，接收窗口和 RT, 抓包分析 TCP 的传输性能。

实验准备

准备两台机器，一台作为服务端，一台作为客户端。

  • 启动服务
  • 服务端、客户端默认系统配置

操作一：默认配置，默认延迟

基于 curl 和抓包信息，可以知道：

![[Pasted image 20240319191920.png]]

  • 三次握手时 iRTT 0.29ms
  • 数据包的 RTT 为 0.1ms
  • 客户端接收窗口是 64240 bytes，window scale factor 为 7

查看 Expert Info， 有四类信息：

![[Pasted image 20240319192150.png]]

  • 蓝色：通信流程的正常信息
  • 青色：一些值得注意的问题，比如重传，重复确认，HTTP 404等
  • 黄色：警告，比如 Full Window
  • 红色：错误信息

传输时序图

时序图几条线的含义：

  • 绿线：接收窗口大小
  • 黄线：已确认的包的序列号
  • 蓝色点：发送的数据包 Segment
  • 红线：代表SACK，表示该红线之前的包已经收到。出现 SACK一般意味着丢包，通常会伴随着重传。
  • 黄线下方的凸起：有 ACK但黄线没有增长，表示 Duplicated。意味着有乱序或重传。

上面时序图放大之后可以看到红线和黄线凸起:

![Pasted image 20240319194203.png]

时序图显示每隔一段时间，服务端会停止发送数据，并且转折点会有很多的 SACK 和重传。

  • Expert Info 中没有 Full-Window 的警告，并且蓝色线上升后，绿线立即上升，说明客户端不是问题。
  • 黄线上升后服务端会立即发送数据，说明 send buffer 满了，得 ACK 来了之后才能继续发送。尝试过将 send buffer 调到最大，但只是停顿的次数变少了，现象依旧。
  • 因此原因猜想是机器之间的网络传输出现丢包，导致部分包的 ACK 一直回不来，触发重传重新收到 ACK 后才继续发送。

操作二：默认配置，100ms 延迟

三次握手

![[Pasted image 20240319202904.png]]

传输时序图

![[Pasted image 20240319202829.png]]

蓝色线数据发送后，绿色线（Receive window）立即增加了，说明客户端 ACK 的非常快，但数据包迟迟没有发送，可以推测是发送端出了问题。

数据包的发送不再是操作一那样接近平滑的曲线，而是脉冲式的每 1ms 发送一次，说明我们设置的延迟生效了

RTT

![[Pasted image 20240319202842.png]]

  • 三次握手时的 iRTT 为 0.1s，符合预期
  • 传输过程的 RTT 基本在 01 ~ 0.2s之间。

之所以出现这种情况，和 Linux 包发送流程以及 tcpdump 的工作机制有关。Linux 的网络栈工作流程大致如图：

![[Linux network stack.png]]

  • 图片来自《System Performance, 2th》

我们通过 tc 设置的延迟，会让网络包在 qdisc 的队列中进行排队，等待 100ms 后再发送到网卡。而 tcpdump 抓包是基于 BPF 来的，BPF 直接过滤网卡的数据包，将其发送到用户态的 tcpdump

![[Pasted image 20240203115254.png]]
图片来自 What is/are TCPdump?

所以对于 tcpdump 抓到的包，RTT 的计算过程是：

  • 三次握手时，先收到客户端 SYN，服务端响应的 SYN+ACK 在 qdisc 排队 100ms 后发送到客户端，因此 iRTT 为 0.1s。

  • 其他服务端发送的数据包，经 qdisc 延迟后进入网卡时被抓包，客户端没有设置延迟，其 ACK 包会很快返回，所以计算出的 RTT 依然是几毫秒。

Linux 更详细的收发包流程可以参考下面几篇文章：

  • Linux Network Stack: 网络包的流转
  • Linux 网络栈接收数据（RX）：原理及内核实现
  • [译] Linux 网络栈监控和调优：发送数据

操作三：wmem 4096，默认延迟

传输时序图

![[Pasted image 20240320195341.png]]

这次是一条非常平滑的曲线，说明缩小 send buffer 并没有影响数据的传输。因为内网传输客户端的 ACK 很快就发回来了，因此不影响服务端的持续发送。这次抓包在操作一抓包完成后进行的，没有出现操作一的情况，也侧面证明第一次抓包应该是机器之间的网络传输出现了问题。

操作四：wmem 4096，100ms

传输时序图

整体看也是一条平滑的曲线

![[Pasted image 20240320203817.png]]

但放大后看每个包的发送有明显 0.1s 的延迟，操作三是没有的，对比如下：

![[Pasted image 20240320203009.png]]

![[Pasted image 20240320202812.png]]

同时和操作 2 相比，因为其 send_buffer 太小，导致一次发送的数据少，加上 100ms 延迟，整体传输时长明显增加。

因此当 RTT 确定时，在没有触发拥塞控制的情况下，可以基于 BDP 适当的增加 send_buffer ，提高 TCP 的传输性能。

操作五：rmem 4096，默认延迟

三次握手信息，客户端的 recv win 为 1460，scale factor 为 2，实际 win = 1460 * 2^2 = 4096，符合我们的设置。

![[Pasted image 20240320204647.png]]

Expert Info 显示有巨多的 full window。

![[Pasted image 20240320204605.png]]
传输时序图

蓝色线填满绿色线，说明数据发送后 recv win 满了，收到 ACK 后 recv win 才继续增加，说明客户端的接收窗口可能会是性能瓶颈。

![[Pasted image 20240320205214.png]]

操作六：rmem 4096，100ms 延迟

整体传输速度巨慢，要 20 几个小时才能下载完，整体传输情况和操作五类似，区别在于这里的是每 100ms 发送一次数据。
